# Bilibili 动态 ZIP 文件名设计

## 目标

下载单条动态原图 ZIP 时，让文件名可以直接识别发布时间、UP 主和内容，同时保持排序稳定、避免重名，并兼容 Windows 文件名规则。

## 文件名格式

```text
YYYY-MM-DD_HHmm_UP主名_动态标题摘要_动态ID.zip
```

示例：

```text
2026-07-15_2036_UP主名_动态标题摘要_1218640478597021717.zip
```

字段规则：

- 时间使用动态发布时间，按 Bilibili 使用的中国标准时间（UTC+8）格式化为 `YYYY-MM-DD_HHmm`，不使用下载时间。
- UP 主名最多保留 30 个字符。
- 优先使用动态正式标题；没有正式标题时，使用正文第一行。
- 标题压缩连续空白后，最多保留 40 个字符。
- 动态 ID 始终保留，作为同一分钟内同名内容的唯一标识。
- 字符长度按 Unicode 字符计算，不能从代理对中间截断表情符号。

## Windows 兼容处理

UP 主名和标题进入文件名前统一处理：

- 将 `<>:"/\\|?*` 和控制字符替换为空格。
- 将连续空白压缩为一个空格。
- 去掉首尾空格，以及 Windows 不允许出现在名称末尾的句点。

不做拼音转换，不移除正常中文、英文或表情符号。

## 数据来源

不新增 Bilibili API 请求。

1. 优先复用脚本已经读取的 `window.__INITIAL_STATE__`，从当前动态详情数据中取得 UP 主名、发布时间、正式标题或正文。
2. 页面内置数据缺少某个字段时，只在当前动态详情容器内读取页面已经显示的对应内容作为兜底。
3. 如果经过两层读取后仍缺少发布时间、UP 主名或标题正文中的任一项，使用现有文件名：`bilibili-dynamic-<动态ID>.zip`。不使用下载时间或“未知 UP 主”等虚构占位词。

元数据只在用户点击下载时读取一次，与图片地址一起用于本次 ZIP，不增加后台轮询。

## 下载流程

现有下载流程保持不变：

1. 收集并下载原图。
2. 生成 ZIP。
3. 生成一次最终文件名。
4. 自动下载和绿色 `保存 ZIP` 备用链接共用该文件名。

不增加设置界面、命名模板配置、网络依赖或新文件。

## 错误处理

文件名生成失败不能阻止图片下载。任何元数据解析或格式化异常都回退到 `bilibili-dynamic-<动态ID>.zip`，其余错误处理沿用当前逻辑。

## 验证

在现有 `checks/self-check.js` 增加最小检查：

- 完整元数据生成预期文件名。
- 非法字符和多余空白被清理。
- UP 主名与标题按字符数截断，表情符号不被截坏。
- 缺少必要元数据时回退到现有文件名。

修改脚本后继续运行：

```powershell
node --check .\bilibili-dynamic-originals.user.js
node .\checks\self-check.js
```

最终在星愿浏览器的真实单条动态页面验证：自动下载得到新文件名，绿色备用链接的 `download` 文件名与其一致。

## 不做

- 自定义命名模板。
- 下载时间选项。
- Bilibili API 请求。
- 批量动态命名或目录分组。
- 修改 ZIP 内部图片名称。
